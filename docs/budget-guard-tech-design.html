0实现状态 · 语言 B · 分工 · 测试方案

语言决定:B = Node(ESM,混合态)

守卫核心从纯 bash 迁移到 Node 22+(纯 ESM)。原因:hook 不锁语言(type:command 跑任意可执行,已源码核实 schemas/hooks.ts:33);MCP server 必须 Node(官方 SDK);统一一种语言贯穿 probe / hook / MCP,且更贴 CodexBar 的健壮取数。bash 旧实现保留并存作 fallback。

  • probe / hook / installer:纯 Node 内置模块,零外部依赖(node:fs/https/crypto/...)。
  • MCP server:Node + @modelcontextprotocol/sdk + zod(唯一带 npm 依赖的组件;安装时需 npm install)。
  • hook 入口仍是 shell command,指向 node ~/.budget-guard/bin/guard.mjs <agent> <phase>。

1目标场景 & 诚实边界

长任务(自主循环)会在 5 小时滚动窗 / 周限额跑到一半被硬切,进度丢失。本项目把"随机硬切"变成"阈值处干净停 + 刷新后续接"。

不能绕过限额。额度耗尽 API 必拒,没有任何手段让 agent"用完还继续"。本方案只保证在耗尽之前停在干净点,并在窗口刷新后续接。
强制 vs 协作。真正强制的只有 PreToolUse deny 与 Stop 的 continue:false;"写 checkpoint""调 park 工具"是协作式——靠协议提示 agent 执行,非框架强制。

2「睡前设 Goal」最终答案

你设个 Goal 去睡觉,机器开着、无人操作:

  1. 跑到 T3(默认 92%) → 守卫让 agent 停止新工作,并调用 wait_until_budget_refresh() 原地 park(进程不退、不耗 token)。
  2. 几小时后 5h 窗口刷新 → 该工具检测到额度回落,返回。
  3. agent 在同一会话、同一上下文里拿到返回,自动接着干——无外部注入、无 tmux、无 watchdog。
前提:进程活着(机器开,你只关心开机情况)。若进程意外死/配置错/机器睡 → C3 兜底(外部 watchdog 重新 --resume)。跨 Claude 和 Codex 都成立。

3续接架构:C4 主 / C1 备 / C3 兜底

路径角色机制上下文 / token适用
C4 阻塞 MCP 工具 跨两家主路 agent 调 wait_until_budget_refresh(),阻塞轮询到刷新返回,同 turn 续 同会话不重放;park 期零 token(续接本身正常花) 进程活着(机器开)
C1 CronCreate Claude 专属可选 守卫停时排"刷新后"一次性同进程 cron,fire 注回 REPL 同 session 不重放 仅 Claude,进程活着
C3 外部 watchdog 必备兜底 launchd 按 reset 调度 → claude --resume / codex exec resume;或 tmux send-keys 注入活 TUI 磁盘 transcript 重放(token 尖峰) 进程死 / timeout 配错 / 机器睡 / headless
C2 云端 Routine 本期不做 Claude 云端 /schedule,全新 session + 仓库 checkpoint 靠仓库 checkpoint (机器关机才需要;Codex 无)
为什么 C4 当主路:它绕过了"外部往任意终端 TUI 注入输入"这个物理难题(进程独占 PTY,外部塞不进),改由 agent 自己调工具 park;还补上了 Codex 没有任何原生定时调度的短板(C1 只 Claude 有)。

4C4 机制详解:阻塞 MCP 工具

守卫(hook) budget MCP server (stdio) 额度端点 │ util≥T3 │ │ │ deny 工具 + 提示 │ │ │ "请调 wait_until_refresh" │ │ agent ──────调用工具─────────────▶│ 轮询 usage ────────────────▶│ │ (turn 挂起等待) │ util 仍高 → 阻塞(每 N 秒查) │ │ ⏳ 零 token │ ……几小时…… │ │ │ reset_epoch 过 / util 回落 │ │◀────────工具返回 {refreshed}──│ │ agent 同 turn 继续干活 ──────────▶ (新窗口,正常消耗)

5强停机制(T3 / 95%)

  • Stop continue:false 可靠终止 query loop:claude query.ts:1267-1280(stop_hook_prevented,不可绕过);codex turn.rs:547(自然 Stop 点)。
  • PreToolUse deny 所有 hook 可见工具(deny 只拒单次、agent 可换工具,故必须全拒)。Codex 现版覆盖 Bash + apply_patch + MCP + 扩展工具(apply_patch.rs:450),不再只 Bash。
  • 精确放行写 checkpoint:仅 file_path 精确等于 checkpoint 路径的 Write/Edit/MultiEdit(及 Codex 的 apply_patch);删除对命令文本的 basename substring grep(旧 :167 漏洞)。
  • Codex 额外 pause goal:codex /goal 是 idle 续跑(MaybeContinueIfIdle goals.rs:137),停后会一空闲就自续 → 硬停须 pause active goal(优先 pause 而非 clear,保目标)+ 持续 deny 兜住。pause 机制按路径:TUI 走 tmux send-keys "/goal pause";headless 无 TUI → app-server 控制或"pause 不可用则持续 deny"兜底。

6取数层(照 CodexBar 逻辑用 bash+jq 重写)

不照搬 Swift(CodexBar 是 1003 文件 GUI app),而是按其逻辑重新实现 + 用其测试 fixture 当校验 oracle。

  • provider 契约:provider_<name>_fetch 统一返回归一化 JSON,guard / watchdog / MCP server 共用同一份(消除现有重复实现)。
  • 双 max:warn_util(所有有 util 的滚动窗桶取 max,含无 reset 的)驱动全部三档(T1/T2/T3)——保守起见,任何桶满都该提醒/停;hard_util(只含 reset 可解析的桶)仅用于 burn-rate 预测和"刷新时间"展示。extra_usage(月度溢出额度,语义不同)只进 doctor、不驱动任何档。语义修正早期设计曾让 T3 用 hard_util,但当非 resettable 桶(如 opus 周限额未带 reset)是瓶颈时 hard_util=0 会漏判强停;cross-review 实证后统一为 warn_util 驱动三档,pre/post/stop 一致。
  • 全 bucket:Claude 遍历 five_hour / seven_day / seven_day_opus / sonnet / oauth_apps / routines / extra_usage(每个 {utilization 0-100, resets_at ISO});Codex 遍历 rate_limit.primary/secondary_window + additional_rate_limits[]({used_percent, reset_at epoch, reset_after_seconds})。
  • reset 解析:Claude ISO→epoch 用 jq fromdateiso8601(去小数秒);Codex reset_at epoch。删除 v≤1 则×100 启发式(会把 0.8% 误放大成 80%)。
  • 端点:Claude GET api.anthropic.com/api/oauth/usage(anthropic-beta + User-Agent: claude-code/<ver>);Codex GET chatgpt.com/backend-api/wham/usage(ChatGPT-Account-Id 取自 ~/.codex/auth.json tokens.account_id;尊重 config.toml chatgpt_base_url)。

7三档守卫 + doctor

T1 · 提醒一次 80%

窗口内只提醒一次。BUDGET_WARN_ONCE

T2 · 每次提醒 90%

持续催收尾 + 写 checkpoint。BUDGET_WARN_REPEAT

T3 · 强停/park 92%

deny + continue:false + 触发 C4/C1。BUDGET_HARD(激进可调 95)
  • 兼容:BUDGET_SOFT → WARN_REPEAT deprecated alias;校验 0≤ONCE<REPEAT<HARD≤100,runtime 非法回退默认。
  • T1 "一次" 用窗口指纹 = agent|scope(pwd -P)|赢家bucket|reset_epoch|档名,存 STATE_DIR/notified/<agent>_<scope>.json;只按 reset_epoch 过期 prune(不用 util<T1,避免边界抖动反复提醒)。

doctor(budget_guard.sh <agent> doctor,非 fail-open)

打印:token 来源 / endpoint / HTTP 状态 / JSON 合法性 / 全 bucket 表(raw+normalized util、reset 解析) / 赢家桶 / 生效阈值(+alias 来源) / MCP server 注册 & codex tool_timeout_sec 是否配够 / hook 配置 / state 目录。退出码 0 健康 1 警告 2 auth/网络 3 schema 4 配置非法。可测:BUDGET_USAGE_FIXTURE + BUDGET_NOW_EPOCH。

doctor 是防"静默失效"的元工具:fail-open 会把"端点/字段错"伪装成"守卫安静正常"。但 parser 必须同补丁修——doctor 只暴露,parser 修了才真保护。

8.5已验证范围 & 限制(真机 E2E 结论 · 诚实边界)

已真机 E2E 验证(可依赖):
  • Claude 守卫全套:pre 硬线 deny、pre 精确放行 checkpoint、stop 硬停(写 pending)—— tmux 真 claude -p 实测。
  • C3 watchdog 续接(双家):claude --resume + checkpoint STEP 1→2;codex exec resume + secret + thread_id 一致 —— 两家都证了上下文连续(非新会话)。这是可靠的无人值守续接路径。
  • doctor 双家:真端点全 bucket 解析(claude seven_day / codex primary_window + additional_rate_limits[Spark])。
  • Codex 守卫(交互 TUI):TUI 里 PreToolUse 真触发 + deny + marker 不生成。
  • 解析 / 三档 / fingerprint / checkpoint 安全 / cache CAS:85 单测 + 10 MCP 单测全过。
已知限制(不要声称覆盖):
  • Codex 守卫仅交互 TUI 可用。codex exec(headless,如 0.135.0)不触发 lifecycle hooks——hooks.json/config.toml 均试、feature enabled、bypass-trust、关 unified_exec 都无效。所以 headless codex 自主任务得不到 pre/post/stop 守卫(但 C3 watchdog 续接仍可用)。
  • C4(阻塞 MCP 工具 park)= 交互/experimental。机制已验(直接调 + SDK E2E:park→flip→ready),但真 agent 在 headless 调用两家都不通(claude -p 不调长阻塞工具;codex exec 在 handler 前 user cancelled)。C4 的真实用途是交互会话;headless 续接走 C3。
  • codex exec 跑 MCP 工具需顶层 --ask-for-approval never(非交互默认取消)。
  • out-of-cwd 绝对 checkpoint 的祖先目录 symlink 不查(非默认配置);cache CAS 未做多进程压测;Node 侧 burn-rate 未实现(bash 专属)。

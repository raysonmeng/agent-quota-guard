1需求场景评估

问题真实:5 小时滚动窗 + 周限额会半路硬切 /goal 这类自主循环,造成进度丢失。把「随机硬切」变成「轮末干净暂停 + 续接」是合理诉求。

最该肯定的一点 —— 它没吹能绕过限额,只承诺「提前停在干净点 + 续接」。诚实且有价值。

两点要清醒:① 硬线等于系统性放弃最后一段额度(安全垫);② 软线只是建议、agent 可无视,真正强制的只有 PreToolUse deny 和 Stop 的 continue:false。

2架构评价

共用一份核心脚本(两包字节相同)+ agent/phase 双参数分派 + fail-open + 统一 exit 0 + JSON。对 hook 环境是稳的取舍,工程品味好,无异议。

4CodexBar 给的「标准答案」

参考实现 ~/repo/CodexBar(Swift menu-bar app,监控各家订阅用量)的取数源码,把两个未知都钉死了:

Claude · GET https://api.anthropic.com/api/oauth/usage URL 对、解析不全

  • 多个桶,每个 = { utilization: number(0-100), resets_at: ISO8601 字符串 }:five_hour、seven_day、seven_day_opus、seven_day_sonnet、seven_day_oauth_apps、seven_day_routines(或 cowork)、iguana_necktie、extra_usage。
  • 桶用到才填对象,否则 null(实测本账户 sonnet 活跃、opus=null)。→ 必须遍历所有桶取 max。
  • 请求还应带 User-Agent: claude-code/<version>(脚本漏了);headers 含 anthropic-beta: oauth-2025-04-20。

Codex · GET https://chatgpt.com/backend-api/wham/usage 我们整段猜错

  • 可被 ~/.codex/config.toml 的 chatgpt_base_url 覆盖(含 /backend-api → /wham/usage,否则 /api/codex/usage)。
  • Header:Authorization: Bearer <tokens.access_token> + ChatGPT-Account-Id: <tokens.account_id> + Accept: application/json(token/account_id 取自 ~/.codex/auth.json)。
  • Schema(与脚本假设完全不同,真机 200 验证): 
    {
  "rate_limit": {
    "primary_window":   { "used_percent":int, "reset_at":epoch, "limit_window_seconds":int, "reset_after_seconds":int },
    "secondary_window": { ... 同上 ... }
  },
  "additional_rate_limits": [        // 模型分桶,如 GPT-5.3-Codex-Spark
    { "limit_name":str, "metered_feature":str, "rate_limit": { primary_window, secondary_window } }
  ]
}
    正好印证「Codex 也有 5h(primary)+ weekly(secondary)双窗」。reset_at 是 epoch,字段名与 Claude 的 ISO resets_at 都不同。

5新设计:三档可调守卫

把现有两档(SOFT 78/HARD 88)改成三档、全部用户可调,把「提醒频率」做成阶梯,把激进度的选择权还给用户。

T1 · 提醒一次

80%
本额度窗口内只提醒一次,不烦人。
BUDGET_WARN_ONCE

T2 · 每次提醒

90%
持续催促收尾、写 checkpoint。
BUDGET_WARN_REPEAT

T3 · 强制停止

92%
pre deny 新工具 + stop continue:false。
BUDGET_HARD(激进可调 95)
为何 T3 默认 92 而非 95:守卫离散采样,一轮可能吃好几个百分点;Codex 侧 hook 只可靠拦 Bash,超调风险更大。默认应兑现「减少硬切」承诺,而非榨满额度;想激进的用户文档给 BUDGET_HARD=95。

兼容:BUDGET_SOFT 设为 deprecated alias,仅当 WARN_REPEAT 未设时映射给它(旧 SOFT 语义=每次软提醒)。

6锁定的实现清单

A · 解析层重构(可测是核心)

  • 拆 fetch_usage_json(agent)(只 curl)+ parse_usage(agent,json,now)(纯函数,出 {ok,util,reset_epoch,reset_after_seconds,bucket_id,buckets[]},可喂 fixture)。
  • 双 max:
    • hard_max(驱动 T3)= 仅「有 utilization 且 reset 可解析」的滚动窗桶取 max。
    • warn_max(驱动 T1/T2)= 所有有 utilization 的滚动窗桶取 max(reset 坏也算)。
    • extra_usage 只进 doctor,不驱动任何档(月度溢出额度,语义不同,计入会误停)。某 seven_day* 桶 reset 解析失败 → 进 warn_max、不进 hard_max + doctor 标警告。

B · reset 类型

  • ISO→epoch 全用 jq fromdateiso8601(去小数秒,先认 Z,异常时区 → doctor 警告);不用 bash date(BSD/GNU 坑)。
  • Codex reset_at(epoch)权威;reset_after_seconds 仅显示/fallback,绝不进 fingerprint(每次都变)。

C · utilization 标度

  • 删掉 v≤1 则 ×100 启发式(会把真实 0.8% 误放大成 80% → 误强停)。当 0–100 用,显示取整。
  • doctor:若 Claude 所有 raw util ∈ (0,1] → 报标度漂移警告,不静默猜。

D · 三档 + env + 兼容

  • 默认 WARN_ONCE=80 / WARN_REPEAT=90 / HARD=92;BUDGET_SOFT→WARN_REPEAT alias。
  • resolve_thresholds() 校验 0≤ONCE<REPEAT<HARD≤100;runtime 非法 → 静默回退默认;doctor 显式报错。

E · 各 phase 行为

phase行为
prehard_max≥HARD → deny(放行规则见 H)
post / stophard_max≥HARD(仅 stop 强停 + 写 pending) → warn_max≥WARN_REPEAT 每次提醒 → warn_max≥WARN_ONCE 指纹门控提醒一次
promptburn-rate「还能跑多久」目标线指向 T3
resume不变(注入 checkpoint 续接)

F · fingerprint「提醒一次」状态

  • fp = hash(agent|scope|bucket_id|reset_epoch|"warn_once");scope = hash(pwd -P)(+ session 可选)。
  • 存 STATE_DIR/notified/<agent>_<scope_hash>.json;只按 reset_epoch 过期 prune + 文件大小封顶;不用 util<T1 清除(边界抖动会反复提醒)。
  • bucket 切换 → 新指纹 → 每个限额各得一次 T1(intended)。

G · doctor 自检 · budget_guard.sh <agent> doctor

  • 非 fail-open。打印:生效阈值(+alias 来源)、token 来源、endpoint URL、HTTP 状态、JSON 合法性、全 bucket 表(raw+normalized util、reset 解析结果)、赢家桶、hook 配置形状、state 目录、当前 fingerprints。
  • 退出码:0 健康 1 警告 2 auth/网络 3 schema/无候选 4 阈值非法
  • 可测:BUDGET_USAGE_FIXTURE=<file> 绕网络、BUDGET_NOW_EPOCH= 固定时间。
为什么 doctor 是头号产品价值:fail-open 会把 schema 错误伪装成「守卫安静正常」。doctor 让用户能确认「守卫真的看得见我的额度」,而不是默默失效。但 parser 修复必须同补丁,因为 doctor 只能暴露问题、parser 修了才真正保护。

H · checkpoint bypass 修法

  • T3 硬线:Bash 一律 deny;只放行 Write/Edit/MultiEdit 且 file_path 精确等于解析后 checkpoint 路径。删除对命令文本的 substring grep。Codex 侧 apply_patch 本不触发 hook、自然放行。

I · Codex 请求修正 + Claude UA

  • URL 默认 …/backend-api/wham/usage;读 chatgpt_base_url;加 ChatGPT-Account-Id + Accept。
  • Claude 加 User-Agent: claude-code/<version>。

J · install.sh / 协议文本

  • CLAUDE.md / AGENTS.md 协议块 + install help 更新为三档 + doctor 用法;幂等合并逻辑不动。

K · 阶段优先级

  1. P0 同一补丁 parser 重写(双 max + 全桶 + 类型解析) + doctor。
  2. P1 checkpoint 精确匹配;Codex URL/header/schema;Claude UA。
  3. P2 per-session/项目状态文件;burn-rate 窗口回归。
